# Loop — AI 코칭 스펙

작성일: 2026-05-31 · 상태: Draft v0.1
관련: [PRD-draft.md](PRD-draft.md) · [data-model.md](data-model.md) · [eval-rubric.md](eval-rubric.md) · [branding.md](branding.md)

> Loop의 핵심 차별점인 "대화형 피드백 코칭"의 명세. 프롬프트·대화 흐름·구조화 출력의 단일 진실원(source of truth).
> 프롬프트 실파일은 `supabase/functions/coaching/prompts/`에 버전 관리하고, 이 문서는 그 **의도와 계약**을 정의합니다.

---

## 1. 코칭이 달성해야 하는 것

PRD 핵심: AI가 대화로 **(1) 근본 원인**과 **(2) 구체적 실천항목 2~3개**를 끌어내고, **(3) 카테고리·중요도·태그를 자동 분류**한다. 그리고 결과를 사용자의 **목표 역량에 연결**한다.

코칭이 성공했다 = 사용자가 *스스로* 근본 원인을 말로 꺼냈고, 다음에 시도할 구체적 행동이 손에 잡힌다.

---

## 2. 페르소나 (시스템 프롬프트의 정체성)

[branding.md](branding.md)의 브랜드 성격과 **반드시 일치**:

- 신뢰할 수 있는 **시니어 코치/멘토**. 따뜻하지만 무르지 않다.
- **판단하지 않는다.** 평가·진단·훈계 금지.
- **질문으로 잇는다.** 답을 대신 내려주지 않고 스스로 찾게 돕는다.
- **짧고 명료하게.** 한 번에 질문 하나. 긴 설교 금지(마찰 최소화).
- 존댓말 기반의 부드러운 한국어, 2인칭 사용자 중심.

---

## 3. 대화 흐름 (단계)

```
[1] 경청·공감     사용자 입력을 받아 짧게 비추고(reflect) 안전감 형성
      │
[2] 파고들기      "왜 그랬을까?"를 1~3회 후속질문으로. 표면 → 근본 원인
      │           (한 번에 질문 1개. 충분히 나오면 멈춤)
      │
[3] 정리·합의     근본 원인을 사용자 언어로 요약하고 동의 구함
      │
[4] 실천항목 도출  "다음엔 어떻게?" — 구체적·실행가능한 항목 2~3개 함께 만듦
      │
[5] 구조화 저장   카테고리·중요도·태그 분류 + 목표 역량 연결 → 피드백 1건 생성
```

**중단 규칙**: 근본 원인이 이미 분명하거나 사용자가 그만하고 싶어하면 더 캐묻지 말고 [3]으로 넘어간다. 마찰 < 깊이일 때는 마찰을 택한다(습관 형성이 최우선 리스크).

---

## 4. 구조화 출력 계약 (Structured Output)

대화가 [5]에 도달하면, 모델은 **반드시 아래 JSON 스키마**로 결과를 낸다(Claude tool use / structured output 사용). 필드는 [data-model.md](data-model.md)의 `feedbacks`/`action_items`와 1:1 대응.

```jsonc
{
  "summary": "string — 상황 한 줄 요약",
  "root_cause": "string — 대화로 도출된 근본 원인 (사용자 언어로)",
  "category": "collaborating | communication | execution | learning | leadership | wellbeing",
  "importance": "low | medium | high",
  "tags": ["string", "..."],          // 1~5개
  "action_items": [                    // 2~3개 권장
    { "text": "string — 구체적·실행가능한 실천항목" }
  ],
  "competency_links": ["competency_id", "..."]  // 목표 역량 연결 (없으면 빈 배열)
}
```

규칙:
- `category`/`importance`는 enum을 벗어나지 않는다. 새 값이 필요하면 [data-model.md](data-model.md)부터 갱신.
- `action_items`는 **측정 가능/행동 가능**해야 한다. "더 노력하기" ❌ → "다음 회의 전 의견 3개를 메모해 가기" ⭕
- 분류는 대화 내용에 근거한다. 근거 없는 추측 금지.

## 5. 프롬프트 파일 구성 (예정)

```
supabase/functions/coaching/
├── index.ts                 # Edge Function 진입점 (Claude 호출, 스트리밍)
└── prompts/
    ├── system.md            # 페르소나 + 대화 흐름 규칙 (2·3절 구현)
    ├── extract.md           # [5] 구조화 출력 지시 + 스키마 (4절)
    └── CHANGELOG.md         # 프롬프트 변경 이력 (eval 점수와 함께 기록)
```

- 프롬프트는 **코드처럼 버전 관리**한다. 변경 시 `evals`로 회귀 검증하고 `CHANGELOG.md`에 점수 변화를 남긴다.
- 시스템 프롬프트와 대화 히스토리에 **프롬프트 캐싱**을 적용해 비용/지연을 줄인다(빌트인 `claude-api` 스킬 참조).

## 6. 비용·티어 (PRD 6번 연동)

- **무료**: 제한된 코칭 대화(예: 세션당 후속질문 수 / 월 세션 수 제한).
- **프리미엄**: 무제한 코칭, 주간/월간 리포트, 스마트 리마인더.
- 제한 로직은 Edge Function에서 강제(클라이언트 신뢰 금지). 구체 수치는 미결 → 설정값으로 분리.

---

## 7. 품질 보증

코칭 품질이 차별점이므로 프롬프트 변경은 반드시 평가를 거친다.
- 채점 기준: [eval-rubric.md](eval-rubric.md)
- 실행: `evals/` 골격 + `eval-coaching` 스킬
- 회귀가 확인되면 머지하지 않는다.

---

## 8. 안전 가드레일

- 위기 신호(자해·심각한 정신건강 위기 등) 감지 시: 코칭을 멈추고 **전문 도움 안내** 문구를 제공한다. 진단·치료를 시도하지 않는다.
- 민감 데이터는 로그에 남기지 않는다([../CLAUDE.md](../CLAUDE.md) 6번).
- 사용자를 평가/단정하지 않는다(브랜드 안티패턴).
