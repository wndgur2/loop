# Loop — AI 코칭 스펙

작성일: 2026-05-31 · 상태: Draft v0.2
관련: [PRD-draft.md](PRD-draft.md) · [feature-spec.md](feature-spec.md) · [data-model.md](data-model.md) · [eval-rubric.md](eval-rubric.md) · [branding.md](branding.md)

> Loop의 핵심 차별점인 "대화형 코칭"의 명세. 프롬프트·대화 흐름·구조화 출력의 단일 진실원(source of truth).
> 프롬프트 실파일은 `supabase/functions/coaching/prompts/`에 버전 관리하고, 이 문서는 그 **의도와 계약**을 정의합니다.
> 코칭에는 두 모드가 있다: **작성(write)** = 새 피드백 도출 · **회고(retrospective)** = 기존 피드백 되새김(§9).

---

## 1. 코칭이 달성해야 하는 것

PRD 핵심: AI가 대화로 **(1) 근본 원인**과 **(2) 구체적 Takeaways 2~3개**를 끌어내고, **(3) category(=하위 목표)·중요도·태그를 자동 분류**한다.

코칭이 성공했다 = 사용자가 *스스로* 근본 원인을 말로 꺼냈고, 다음에 시도할 구체적 행동(Takeaway)이 손에 잡힌다.

---

## 2. 페르소나 (시스템 프롬프트의 정체성)

[branding.md](branding.md)의 브랜드 성격과 **반드시 일치**:

- 신뢰할 수 있는 **시니어 코치/멘토**. 따뜻하지만 무르지 않다.
- **판단하지 않는다.** 평가·진단·훈계 금지.
- **질문으로 잇는다.** 답을 대신 내려주지 않고 스스로 찾게 돕는다.
- **짧고 명료하게.** 한 번에 질문 하나. 긴 설교 금지(마찰 최소화).
- 존댓말 기반의 부드러운 한국어, 2인칭 사용자 중심.

---

## 3. 대화 흐름 — 작성(write) 모드

```
[1] 경청·공감      사용자 입력을 받아 짧게 비추고(reflect) 안전감 형성
      │
[2] 파고들기       "왜 그랬을까?"를 1~3회 후속질문으로. 표면 → 근본 원인
      │            (한 번에 질문 1개. 충분히 나오면 멈춤)
      │
[3] 정리·합의      근본 원인을 사용자 언어로 요약하고 동의 구함
      │
[4] Takeaways 도출 "다음엔 어떻게?" — 구체적·실행가능한 항목 2~3개 함께 만듦
      │
[5] 구조화 저장    title·category(하위목표)·importance·tags 분류 → 피드백 1건 생성
```

**중단 규칙**: 근본 원인이 이미 분명하거나 사용자가 그만하고 싶어하면 더 캐묻지 말고 [3]으로 넘어간다. 마찰 < 깊이일 때는 마찰을 택한다(습관 형성이 최우선 리스크).

**category 배정 규칙**: 모든 피드백은 **반드시 하나의 하위 목표(category)에 속한다**(필수). AI가 사용자의 하위 목표 집합 중 가장 맞는 하나를 고르고, 모호하면 [3]~[5]에서 **짧게 확인 질문**을 해서라도 하나를 배정한다. 미분류는 없다.

---

## 4. 구조화 출력 계약 (Structured Output)

대화가 [5]에 도달하면, 모델은 **반드시 아래 JSON 스키마**로 결과를 낸다(Claude tool use / structured output 사용). [feature-spec.md](feature-spec.md)의 Canonical Template, [data-model.md](data-model.md)의 `feedbacks`/`takeaways`와 1:1 대응.

```jsonc
{
  "title": "string — 피드백 제목",
  "situation": "string — 상황 설명 (템플릿 ## Feedback)",
  "root_cause": "string — 대화로 도출된 근본 원인 (사용자 언어로)",
  "category": "string — 사용자의 하위 목표 중 하나의 id/name (필수, 미분류 불가)",
  "importance": "high | mid | low",
  "tags": ["string", "..."],          // 1~5개
  "takeaways": [                       // 2~3개 권장, 각 항목은 개별 추적됨
    { "text": "string — 구체적·실행가능한 행동/마음가짐" }
  ]
}
```

규칙:
- `category`는 **사용자의 하위 목표 집합에서만** 고른다. 새 분류를 임의 생성하지 않는다. 적합한 게 없다고 판단되면 대화로 확인하되, 최종적으로 하나를 반드시 배정한다.
- `importance`는 `high | mid | low` 외 값을 쓰지 않는다.
- `takeaways`는 **측정 가능/행동 가능**해야 한다. "더 노력하기" ❌ → "다음 회의 전 의견 3개를 메모해 가기" ⭕
- 분류·요약은 대화 내용에 근거한다. 근거 없는 추측 금지.

## 5. 프롬프트 파일 구성 (예정)

```
supabase/functions/coaching/
├── index.ts                 # Edge Function 진입점 (Claude 호출, 스트리밍, mode 분기)
└── prompts/
    ├── system.md            # 페르소나 + 작성 대화 흐름 규칙 (2·3절)
    ├── extract.md           # [5] 구조화 출력 지시 + 스키마 (4절)
    ├── retrospective.md     # 회고 모드 프롬프트 (9절)
    └── CHANGELOG.md         # 프롬프트 변경 이력 (eval 점수와 함께 기록)
```

- 프롬프트는 **코드처럼 버전 관리**한다. 변경 시 `evals`로 회귀 검증하고 `CHANGELOG.md`에 점수 변화를 남긴다.
- 시스템 프롬프트와 대화 히스토리에 **프롬프트 캐싱**을 적용해 비용/지연을 줄인다(빌트인 `claude-api` 스킬 참조).

## 6. 비용·티어 (PRD §6 연동)

- **무료**: 제한된 코칭 대화(예: 세션당 후속질문 수 / 월 세션 수 제한).
- **프리미엄(v1.1)**: 무제한 코칭, 주간/월간 리포트.
- 제한 로직은 Edge Function에서 강제(클라이언트 신뢰 금지). 구체 수치는 미결 → 설정값으로 분리.

---

## 7. 품질 보증

코칭 품질이 차별점이므로 프롬프트 변경은 반드시 평가를 거친다.
- 채점 기준: [eval-rubric.md](eval-rubric.md)
- 실행: `evals/` 골격 + `eval-coaching` 스킬
- 회귀가 확인되면 머지하지 않는다.

---

## 8. 안전 · 범위

- 사용자를 **평가·진단·단정하지 않는다**(브랜드 안티패턴).
- 민감 데이터는 로그에 남기지 않는다([../CLAUDE.md](../CLAUDE.md) §6).
- **위기/정신건강 상담은 MVP 비범위**(2026-05-31 확정). 코칭은 진단·치료·위기개입을 시도하지 않는다. 관련 정책은 추후 별도 검토([feature-spec.md] 비범위).

---

## 9. 대화 흐름 — 회고(retrospective) 모드

**목적**: 기존 피드백을 되짚어 내재화를 촉진한다(작성과 별개 모드).

- **하위 목표 단위로 진행**: 사용자가 하위 목표 하나를 고르면, 그 그룹의 피드백(특히 **미내재화·high importance**)을 대상으로 대화한다. 피드백을 통합 관리하지 않는다.
- 흐름(잠정): 대상 피드백 환기 → "그 뒤로 비슷한 상황이 있었나요?" → 실행 여부 확인 → **Takeaway `done` / 피드백 `internalized` 갱신** 또는 새 피드백 파생.
- 페르소나·톤은 §2 공유. 압박·죄책감 없이 부드럽게 환기(브랜딩 리마인더 톤).
- ⚠️ 그룹 내 대상 선정 로직(미내재화/오래된 것/importance 가중)은 미확정([feature-spec.md] F9).
