# Loop — 기능 명세 (Feature Spec)

작성일: 2026-05-31 · 최종 수정: 2026-06-07 · 상태: **Draft v0.3 — Loopi 채팅 구조·회고 UX 확정, 일부 UX 미결**
관련: [PRD.md](PRD.md) · [data-model.md](data-model.md) · [loopi-spec.md](loopi-spec.md) · [../CLAUDE.md](../CLAUDE.md)

> PRD를 ground truth로, 사용자 시나리오(2026-05-31 확정)로 구체화한 명세.
> 표기: 〔PRD §n〕 = 근거 · ✅확정 · ⚠️ = 결정 필요 · 🚫 = MVP 비범위.

---

## 0. 핵심 모델 (확정)

- **앱 이름 = Loop** ✅
- **목표 구조**: 최종 목표 **1개** + 그 아래 **하위 목표 다중**. (하위 목표 = 기존 문서의 "역량/competency"를 가리키는 사용자 용어로 통일)
- **category = 하위 목표** ✅ — 고정 분류 enum이 없다. **모든 피드백은 반드시 하나의 하위 목표에 속한다**(필수, 미분류 없음). AI가 "어느 하위목표에 닿는지" 추측하고, 모호하면 대화로 확인해서라도 하나를 배정한다. → 피드백↔목표 정렬이 구조적으로 보장됨(PRD 핵심 가치).
- **Takeaways = 개별 항목 리스트** ✅ — 각 항목마다 실행(done) 추적. 피드백은 별도로 `internalized`(내재화) boolean 보유.
- **두 지표**〔PRD §8〕: 내재화율 = 내재화 피드백/전체 · 실천(Takeaway) 실행률 = done 항목/전체.
- **importance** = `high | mid | low` ✅
- **작성·회고 = 같은 채팅 엔진** ✅ — 모드 차이는 *시스템 프롬프트 + 툴*뿐. 툴은 모드당 1개(작성=피드백 생성, 회고=피드백 수정), context는 둘 다 **전체 피드백**. (↓ Loopi 채팅 구조)
- **위기/정신건강 가드레일** = 🚫 MVP 비범위(2026-05-31 확정).

---

## 1. 사용자 시나리오 (스파인)

1. 입장 → **최종 목표 입력** (ex: "Product Owner 달성") — F2
2. **하위 목표 다중 선택** (AI 추천 + 직접 추가) (ex: collaborating, engineering, product planning) — F3
3. **AI 대화로 피드백 작성 + 내재화**, AI가 category(=하위목표)·tags 추측 — F4, F5
4. **피드백 목록/상세/수정/삭제/직접 생성** — F6
5. **대시보드**로 내재화·현황·추이 한눈에 — F8
6. **회고**: AI와 대화로 지난 피드백을 되새김 — F9

## 피드백 데이터 형태 (Canonical Template)

모든 피드백은 아래 형태로 귀결된다. AI 구조화·직접 생성·수정 모두 이 형태를 따른다.

```markdown
---
created_at: { YYYY-MM-DD }
internalized: { true | false }
category: { 하위 목표 중 하나 }
tags: [{ tag1 }, { tag2 }, ...]
importance: { high | mid | low }
---

# {title}

## Feedback

{상황 설명}

## Root cause

{근본 원인}

## Takeaways

- [ ] {미래에 같은 상황에서 할 구체적 행동/마음가짐 1}
- [ ] {2}
- [ ] {3}
```

- `category`는 사용자의 하위 목표 집합에서 선택된 **단수·필수** 값(FK, NOT NULL).
- `Takeaways`는 **체크 가능한 개별 항목** 리스트(각 항목 done 추적). 행동·마음가짐 모두 가능.
- `internalized`는 피드백 단위. ⚠️ 내재화 판정을 *수동 표시*로 할지 *모든 Takeaway done 시 자동*으로 할지 — 기본안: **수동 표시**(회고에서 갱신 유도). 확정 필요.
- 매핑 상세는 [data-model.md](data-model.md).

## Loopi 채팅 구조 (작성·회고 공통) ✅ — 핵심

작성과 회고는 **별개 화면·별개 엔진이 아니라 같은 채팅 한 장**이다. 모드 차이는 *시스템 프롬프트 + 에이전트가 쥐는 툴*뿐이고, **툴은 최대한 단순하게 모드당 1개**로 둔다. 두 모드 모두 **사용자의 전체 피드백을 컨텍스트**로 받는다.

| 모드 | 탭 | context | 툴(1개) | 결과 |
|------|----|---------|---------|------|
| **작성(write)** | 피드백(홈) | 전체 피드백 | `피드백 생성`(create) | 새 피드백 1건(Canonical Template) |
| **회고(retrospective)** | 회고 | 전체 피드백 | `회고`(피드백 수정/update) | 기존 피드백의 `internalized` · takeaway `done` · 다짐 수정 |

- **context = 전체 피드백**: 별도 검색 툴이 필요 없다. 모델이 과거 피드백을 통째로 보므로 **같은 상황의 반복을 자연히 알아챈다**(반복 감지). 작성 중에도 "이거 전에도 있었네요"를 짚을 수 있다.
  - ⚠️ 피드백이 많아지면 컨텍스트 비용↑ → **프롬프트 캐싱**으로 완화하고(빌트인 `claude-api` 스킬), 규모가 커지면 요약/필터링 도입 검토. (MVP 규모에선 전체 주입으로 충분.)
- **툴은 모드당 1개**:
  - **작성 = `피드백 생성`** — 대화가 무르익으면 Canonical Template 1건을 만든다([loopi-spec.md](loopi-spec.md) §4 구조화 출력 계약 = 이 툴의 입력 스키마).
  - **회고 = `회고`(피드백 수정)** — 한 툴이 내재화 표시·takeaway done·다짐 다듬기를 **파라미터로** 모두 처리한다. 새 피드백을 만들지는 않는다(새 상황은 피드백 탭에서 작성).
- **상태 변경은 확인 후 커밋**: 회고 툴이 변경을 제안하면 대화에 **확인 칩**([내재화로 표시] / [아직요] 등)으로 띄우고, 사용자가 누를 때 반영한다. 에이전트가 "네"만 듣고 조용히 바꾸지 않는다(민감 데이터, [../CLAUDE.md](../CLAUDE.md) §6).
- 페르소나·대화 흐름·중단 규칙·구조화 계약은 [loopi-spec.md](loopi-spec.md)가 단일 진실원(작성 §3·§4, 회고 §9). 모든 호출은 **Edge Function 경유**.

> 이 구조 덕분에 작성/회고 경계가 매끄럽다: 같은 채팅 UI·같은 컨텍스트에 **툴만 바뀐다.**

## F1. 온보딩 & 계정

**목적**: 가입 → 최종 목표·하위 목표 설정 → 첫 피드백 진입.

- 인증: Supabase Auth. ⚠️ **로그인 방식** — 이메일/매직링크 / 소셜(Apple·Google). (iOS 심사상 소셜 제공 시 Apple 로그인 필수 가능.)
- ⚠️ **온보딩에서 목표 설정 강제 여부**〔PRD §11 미결〕. 기본안: 권장하되 건너뛰기 허용.

---

## F2. 최종 목표 (1개)

**목적**: 모든 피드백을 정렬할 단일 커리어 목표.〔PRD §5〕

- 제목(필수) + 설명(선택). MVP는 활성 **1개**〔§5〕(스키마는 다중 허용, 앱에서 제약).
- 수정/삭제 가능.

---

## F3. 하위 목표 (다중)

**목적**: 최종 목표를 구성하는 영역들. 피드백의 `category`로 쓰인다(= 핵심 모델).

- **다중 선택/추가**. 입력 방식: **AI 추천 + 직접 추가** ✅(사용자 시나리오).
  - AI 추천: 최종 목표로부터 하위 목표 후보 제시(ex: PO → collaborating, engineering, product planning …). Edge Function 경유.
  - 직접 추가/이름 변경/삭제 가능.
- ⚠️ **하위 목표 삭제 시** 연결된 피드백의 category 처리 — 기본안: 삭제 차단(연결 피드백 있으면) 또는 "보관" 처리. 확정 필요.
- ⚠️ 하위 목표 **개수 상한** 둘지(UX/대시보드 가독성) — 기본안: 느슨한 상한(예: ~8개).

---

## F4. 대화형 피드백 작성 (Loopi) — 핵심

**목적**: 상황을 말하면 AI가 후속 질문으로 **근본 원인 + Takeaways**를 함께 정리.〔§5, §7〕

- **위치/구조**: **피드백(홈) 탭** 하단의 상시 입력 — 위쪽 피드백 목록과 같은 화면(레이아웃은 회고 탭과 일치). [Loopi 채팅 구조](#loopi-채팅-구조-작성회고-공통) 따름: context = **전체 피드백**, 툴 = **`피드백 생성`** 1개.
- 흐름·페르소나·중단 규칙은 [loopi-spec.md](loopi-spec.md)가 단일 진실원.
- 모든 AI 호출은 **Edge Function 경유**〔CLAUDE.md §6〕.
- 입력 마찰 최소화 최우선〔§10〕 → 텍스트 우선. ⚠️ **음성 입력(STT)** MVP 포함 여부 — 기본안: 제외(v1.1).
- ⚠️ **MVP 대화량 상한**(비용 보호) 둘지 — 설정값으로 분리. (구독은 🚫 v1.1)
- 🚫 위기/정신건강 상담 시도하지 않음(비범위).

---

## F5. 피드백 구조화 (자동 분류)

**목적**: 대화 결과를 Canonical Template 1건으로 변환.〔§5〕

- AI 추측 필드: `category`(=하위목표 중 **하나, 필수**), `tags[]`, `importance`(high/mid/low), `title`.
- 생성 필드: `Feedback`(상황), `Root cause`, `Takeaways[]`.
- 사용자가 **저장 전/후 모두 수정 가능**(자동 분류 보정). category는 하위목표 목록에서 선택.

---

## F6. 피드백 목록 · 상세 · CRUD · 직접 생성

**목적**: 피드백을 모아 보고 개별 관리.〔§5〕 시나리오 4단계.

**목록**

- 최신순 기본. 필터: category(하위목표)/importance/내재화/태그. ⚠️ MVP 필터 범위 — 기본안: 내재화 + category.
- ⚠️ 검색 MVP 포함 — 기본안: 제외.
- 카드: title, category, importance, 내재화 배지.

**상세**: 템플릿 전체 + 연결 하위목표 + Loopi 대화 보기. **수정/삭제**(삭제는 확인 단계).

**직접 생성** ✅: AI 없이 사용자가 템플릿을 직접 채워 피드백 생성 가능(시나리오 4단계).

---

## F7. 내재화 & Takeaway 추적

**목적**: 행동으로 이어졌는지 추적 = 핵심 가치.〔§5, §7, §8〕

- 각 **Takeaway**: done 토글.
- 피드백 **internalized**: 내재화 완료 표시. ⚠️ 수동 vs 자동(F0/템플릿 주석 참조) 확정 필요.
- 지표: 내재화율(피드백) + Takeaway 실행률(항목).〔§8〕

---

## F8. 통계 대시보드

**목적**: 회고를 한눈에, 성장(내재화) 체감.〔§5, §8〕 시나리오 5단계.

- **category(하위목표)별** 분포 — 목표 영역별 피드백 쏠림을 보여줌.
- **importance별** 분포.
- **시간 추이**(작성/내재화).
- **태그 빈도**.
- **내재화율**(핵심 지표, 중심 배치) + Takeaway 실행률.
- ⚠️ 기간 필터(주/월/전체) 범위 — 기본안: 전체 + 월.
- 빈 상태 카피(브랜딩 톤).

---

## F9. 회고 (되새김) — 핵심

**목적**: AI와 대화로 **지난 피드백을 되새겨** 미내재화 항목을 행동·내재화로 잇는다.〔PRD 핵심 가치 3〕 → 메모앱이 못 하는 "같은 실수 반복 차단"이 차별점.

**구조**: 회고 탭은 **피드백(홈) 탭과 레이아웃이 일치**한다 — 상단 카드/콘텐츠 + **하단 상시 채팅 입력**. 채팅 엔진·컨텍스트(전체 피드백)는 작성과 공유하고 **툴만 `회고`(피드백 수정)**. (↑ [Loopi 채팅 구조](#loopi-채팅-구조-작성회고-공통))

**진입 (둘 다 같은 회고 채팅으로 합류)**

- **추천 회고 카드**(상단) 탭 — 앱이 되새길 거리를 골라 떠먹인다(진입 마찰↓ = 습관 형성). 카드를 누르면 그 피드백을 인용하며 대화 시작.
- **직접 입력**(하단) — "요즘 협업이 어떤지 보고 싶어"처럼 말로 범위를 좁혀 시작. 새 상황을 적으면 전체 피드백 컨텍스트로 **반복을 감지**해 옛 다짐을 끌어온다.

**추천 카드 (= 대상 선정 로직, 백엔드 쿼리)**

- 카드는 대화 시작 *전*에 떠야 하므로 에이전트 툴이 아니라 **서버 쿼리**가 만든다. 대화 중 추가 환기는 모델이 전체 컨텍스트에서 직접.
- 카드 종류(잠정): **오늘의 되새김**(미내재화·high 단건) · **또 반복됐어요**(같은 영역/태그 반복) · **영역 통째 되짚기**(하위목표 단위).
- ⚠️ 랭킹 세부(미내재화 우선 / 오래됨 / importance 가중 / 반복 가중의 결합)는 미결 — 진입·구조는 확정, 가중치만 추후 튜닝.

**결과**: 회고 툴로 해당 피드백의 `internalized`·takeaway `done`을 갱신하거나 **다짐을 다듬는다**(안 먹힌 Takeaway를 더 구체적으로 수정). 변경은 확인 칩으로 커밋. 새 피드백 생성은 회고 범위 밖(피드백 탭에서).

**톤**: 압박·죄책감 없이 부드럽게 환기. "아직 못 했다"도 이탈로 잇지 않는다([loopi-spec.md](loopi-spec.md) §2·§9).

> 〔변경〕 이전 안의 "회고는 하위목표 단위로 그룹 진행"은 **전체 피드백 컨텍스트 + 추천 카드**로 대체된다. 하위목표 단위 되짚기는 "영역 통째" 카드 한 종류로 남는다.

## 비범위 (MVP 제외 · 기록용)

- **프리미엄 구독·주간/월간 리포트·기기 간 동기화·데이터 내보내기** 🚫 v1.1〔§6, §9〕.
  - ⚠️ 긴장: PRD는 클라우드 동기화를 프리미엄으로 두나 Supabase는 기본 동기화 → 구독 설계 시 재검토.

## 화면 인벤토리 (MVP, 잠정)

```
온보딩: 소개 → 가입·로그인 → 최종 목표 입력 → 하위 목표 선택

메인 (하단 탭 4개):
[1] 피드백(홈)  상단: 현황(내재화율)+피드백 목록 · 하단: 작성 채팅 입력
               └ 항목 탭 → 피드백 상세 → 수정/삭제 · 직접 생성(템플릿)
[2] 회고        상단: 추천 회고 카드(오늘의 되새김·반복·영역) · 하단: 회고 채팅 입력
[3] 대시보드    내재화율 중심 통계
[4] 설정        계정, 최종/하위 목표 관리

* 1·2 탭은 레이아웃 일치 — "상단 콘텐츠 + 하단 상시 채팅 입력". 작성/회고가
  같은 채팅 엔진이라 조작감도 동일(툴만 다름). → Loopi 채팅 구조.
```
